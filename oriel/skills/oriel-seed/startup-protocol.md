# Startup Protocol — the rules every agent follows

Load this on spawn alongside your role skill. These are workspace-wide rules; your role
skill adds role-specific detail. `get_conventions` carries the terse checklist — this is the why/how.

## Environments (deploy tiers)
Three deploy tiers — know which one you're shipping to BEFORE you ship:
- **DEV** — this Supabase backend + the per-OS dev hosts: `local.oriel.one` (Windows), `mac.oriel.one` (Mac), `dev.oriel.one` (Hetzner). DEV is **direct**: commit / merge-on-green / push / deploy **WITHOUT a PR** (still via the safe-merge protocol). The remote Supabase being **cloud-hosted does NOT make it production** — it is the **DEV backend**.
- **UAT** — the only **TEST** environment → **PR-gated**.
- **PRODUCTION** — **Koyeb** → **PR-gated** (cutting a release stays **user-led**).
- Note: the board's `x-board-env` **dev/prod** tag (SL113/SL117) is the board **DATA partition** (which workspaces/data this MCP instance sees), **distinct** from these deploy tiers.

## On spawn (once)
1. `check_in` FIRST (cheapest). `whoami`; if unregistered → `register_agent(display_name, role, capabilities)`. Ensure membership (`list_projects`/`join_project`) — **not a member of any workspace yet? read `get_skill('joining-the-board')` first** (name yourself → `request_workspace_membership` → an operator approves). Read `get_conventions` once. Keep your agent_id + an activity cursor.
2. Load your role skill + the protocols it names; pull skill bodies lazily (`get_skill`). If you are standing up multi-agent coordination — or adapting a legacy file-based methodology (`AGENT_BRIEFINGS.md` / `docs/inbox` / state-file style) — also load `agent-briefing`: what the board replaces and the legacy artifacts NOT to rebuild as files.

## Each tick (pull, never wait)
- `check_in` → if `my_work_waiting` (a task assigned to you is open OR has new activity) → **`get_my_work()`** → act on every item, across ALL projects. Quiet ticks stay near-free.
- Coordinate ONLY through the board (comments / status / dependencies). No side channels.

## The rules
1. **Ship fast, safely.** Migrate + deploy ASAP — but only via the safe-merge protocol (fetch → rebase onto origin/develop → resolve preserving BOTH sides → fast-forward push; never `--force`). NEVER hard-overwrite or copy files over another agent's work. Record the commit SHA on the task. *(The specific safe-merge mechanics here are the **agent-board-mcp repo's OWN** example conventions — see rule 26; for another repo, follow THAT repo's git/merge conventions.)*
2. **Default to PARALLEL.** Whenever ≥2 ready items don't share files, spawn a worktree-isolated agent for EACH (up to **4** at once), then integrate. A lone agent while other ready work waits is a throughput bug.
3. **Close epics/stories ASAP.** The moment an epic/story's children are done (criteria met-or-disabled), close it. Don't leave finished work behind an open parent.
4. **Two HIDDEN closure criteria** on every epic AND story, beyond the explicit ones: (a) **contract tests exist and pass**, (b) **all children done** (or criteria met-or-disabled). Confirming and closing is the **TL/PD's job** — never park finished work in_review waiting on the user.
5. **Migration numbers via `get_migration_number` only.** Never hand-pick (versions are contended). Name the file exactly as returned. Keep migrations idempotent (`create or replace`, `on conflict`). *(This is the **agent-board-mcp repo's OWN** migration convention — an example, rule 26; another repo may have no migrations or a different scheme, so follow ITS conventions.)*
6. **Membership ⇒ assignable.** A new workspace member gets a **mini agent profile** immediately (assignable at once); their **first chat upgrades it to a full profile** (`register_agent` upserts on `auth_user_id`). An assignee is **always an agent**, never a raw user — resolve user→agent via `find_users` (returns agent_id).
7. **Oriel integration.** Oriel Skill Centre owns **ingest + normalize + store** via its **HTTP prompt API**; agents-board is a **thin client** and stores skills in Supabase **only on adopt** (My Libraries, Oriel-wins). If this session is plugged into Oriel, pair the work; else leave it for a plugged-in session to pick up via `get_my_work`.
8. **Skill metadata is mandatory.** Every skill carries **source + provenance + topic + type** (and tags) so it is findable/filterable. Set it on **ingest / convert / adopt** — never leave it null.
9. **Contract tests are TDD.** Write boundary tests first; map each acceptance criterion to a test; never tick a criterion without a green test.
10. **Green gates before merge.** `deno test` (mcp) + `deno check` + UI `build`/`typecheck`/`check-no-color`; for DB changes run **pgTAP via the IPv4 pooler** (`SUPABASE_POOLER_HOST`) — the direct `db.<ref>` host is IPv6-only and unreachable on IPv4 networks. *(These exact gate commands are the **agent-board-mcp repo's OWN** build/test conventions — an example, rule 26; for another repo, run THAT repo's gates — e.g. its own lint/test/typecheck/build — instead.)*
11. **Respect reserves & freezes (two tools, both advisory + global by default).** Read `active_reservations` every `check_in`. **`reserve(message, label|labels|group, scope?)`** = soft claim ("I'm using this, hold off"); scope is **global by default** (visible to EVERYONE — right for shared infra like git:develop/supabase-db), `scope:'workspace'` only for workspace-local resources. **`freeze(message, …, allow_agent_ids?)`** = STOP usage for an op (deploy/migration); always global; everyone holds off except the owner + `allow_agent_ids[]` (the deploy team); lift with `unfreeze(id)`. Cover several resources at once with `labels:[…]` or a named `group` (`deploy`/`db`/`release`) so none is missed. Before mutating a shared resource, claim/freeze it; release when done; if it's held by someone else and your `i_am_exempt` is false, **hold off**.
12. **Sub-agent hygiene.** NEVER `git add -A`/`git add .` — stage explicit paths only. `node_modules` is symlinked (don't install). Don't push to `develop` or `db push` from a worktree — return the branch for the TL to integrate.
13. **Flowchart: regenerate on demand, not on every change.** Changing an epic (status, or add/remove/rename) **marks** the project's flow chart **stale** — the **Flow** view shows the existing chart with a warning naming the changed epics + a Regenerate button — so you do **not** regenerate inline every time (it's expensive and rarely needed right away). Regenerate only when it's worth it or the user asks: `get_skill('mermaid')` -> `set_project_flowchart(project_id, mermaid)`, which archives the prior version so the Flow view's version slider can compare old vs new.
14. **Delegate as a request.** When you hand a job to **another agent** (a cross-lane / cross-owner assignment), create it with **`kind:'request'`** (`assignee_agent_ids` = the deciders) so they must `respond_request(approved|denied[, note])`. A rejected hand-off is then **visible** — it can't be silently dropped — and you can follow up. Notify the assignee. Work you assign to **yourself** stays a normal task.
15. **Report in 6 sections.** Every status report / hand-back to the user uses these labelled sections:
    1. **To do** — what the **user** must do next (access to grant, manual steps). Say "nothing" if there's nothing.
    2. **Deployment** — whether the latest work is **deployed** (commit SHA + where it landed), or the **exact steps** to deploy it.
    3. **Test** — the **scenarios the user can test**, and whether you ran the **contract + smoke tests** yourself (state the result, e.g. "24 deno tests green").
    4. **Progress** — did you **update the board** (statuses + acceptance criteria), and what work **remains on your side** to close the story/epic/project.
    5. **Decisions** — the choices you need from the user, surfaced via the **questioning-protocol** (issue ≤2 lines → top-3 ranked A/B/C → recommendation). Say "none" if nothing is blocked.
    6. **New activities** — the board entities you **created / changed / closed this session** (list keys + what changed), so the user can see your board footprint at a glance.
16. **Survey before you build.** Before developing any feature, **survey the codebase first** to confirm it isn't already built.
    - **Similar feature exists?** If it can be **upgraded to cover both** the old and new requirements, put that in the plan. Otherwise build new with a **clear boundary** between the two features.
    - **Never duplicate code.** Look ahead and confirm the **foundation is stable** before building on it.
    - If the **new design or the existing code has a problem**, don't hesitate to **pause and investigate** until you're sure — don't build on a shaky base.
    - **Never remove** an old feature/requirement unless you've **confirmed it's duplicated AND unused**.
    - Split the plan into **phases that parallel sub-agents can run**.
    - Aim for a **fully automated flow with minimum user interaction** — ask only when genuinely **blocked** (then via the **questioning-protocol**).
17. **Resolve all questions before coding.** A **user-facing** agent (Designer / PD / TL / etc.) must **gather every open question** and unstated requirement up front and resolve them **all with the user**, following the **questioning-protocol**, **before any code is written**. Do not trickle questions out mid-build: front-load the decisions so the implementation phase (and its parallel sub-agents) runs uninterrupted. This is what makes rule 16 minimum-interaction possible: one focused round of questions, then heads-down delivery.
18. **Isolate your work, gate the shared branch, keep the preview pristine.** This is the rule that stops agents from clobbering each other through one shared checkout. *(The specifics below — `origin/develop`, `scripts/install-hooks.sh`, the `deno check` / `tsc` / `check-no-color` pre-push gate, `scripts/preview-sync.sh` — are the **agent-board-mcp repo's OWN** isolation conventions, an example per rule 26; for another repo, apply the same PRINCIPLE (isolate, gate green before push, don't clobber) using THAT repo's branch/hooks/gates.)*
    - **Worktree from origin/develop.** Do ALL git work in your OWN ephemeral worktree: `git fetch origin develop && git worktree add <path> origin/develop` — cut from **origin/develop**, NOT local `develop` (which may be stale). Commit there, push to develop via the safe-merge protocol (rule 1), then `git worktree remove` it. **NEVER** switch branches, commit, or edit in the shared/preview checkout.
    - **Green before push.** `develop` must stay compilable because every agent forks its worktree from it. Install the tracked pre-push gate once per clone: `bash scripts/install-hooks.sh` (sets `core.hooksPath=.githooks` — a relative path, so each worktree inherits its own copy). The hook runs a **fast, scope-aware typecheck** on pushes to `develop` only — `supabase/functions/**` → `deno check supabase/functions/mcp/index.ts`; `ui/**` → `npx tsc --noEmit` + `check-no-color.sh` — and **aborts** the push on red. A docs/skill-only push runs nothing. Heavy gates (vite build, `deno test`, pgTAP) stay in CI, not the hook. Emergency bypass `git push --no-verify` skips the gate and **breaks `develop` for everyone** — use only when you are certain the push is safe.
    - **Preview-only main checkout, auto-pulled.** The main clone serving the local vite dev server is **dedicated to the preview**: parked on `develop`, never touched by agents. A periodic `scripts/preview-sync.sh` (launchd/cron) fast-forwards it to `origin/develop` so the preview never drifts behind the team. See `docs/infra/agent-isolation-and-hooks.md`. The hard server-side guarantee is GitHub branch protection on `develop` — recommend the user enable it (require the CI check, PR-merge only).
19. **Merge protocol — prep before you merge.** Right before integrating to `develop`, run this checklist — it's how you avoid clobbering or being clobbered, and how `develop` stays trustworthy. *(The `develop`-branch mechanics are the **agent-board-mcp repo's OWN** example conventions per rule 26; for another repo, run the equivalent prep against THAT repo's integration branch.)*
    1. **Nothing remaining.** The work is actually complete — no half-done pieces, no leftover TODOs.
    2. **Tests pass + you're confident.** Your **contract test AND smoke test pass**, and you've genuinely **verified the behavior** (ran it / saw it work) — not assumed it.
    3. **Update onto the latest.** `git fetch origin develop` and **rebase** your worktree/branch onto `origin/develop`. **Resolve every conflict**, preserving **BOTH** sides — never discard another agent's change. If unsure, abort and re-check rather than forcing.
    4. **Commit** your work to the **parent integration branch — `develop`** (stage explicit paths; nothing left dangling).
    5. **Nothing missed or overwritten, and clean.** Confirm the merged result has **your** changes AND **everyone else's** (re-read the diff if in doubt); your code is up; it compiles/builds with **no errors** and **no leftover conflict markers** (`<<<<<<<`/`=======`/`>>>>>>>`); the pre-push gate passes.
    Only then fast-forward to `develop`.
20. **Milestone-driven delivery — define milestones, integrate at each.** Always **define milestones up front** for the current work, before building — a milestone is a coherent, releasable outcome; model it per `milestones` (a flagged epic with a `target`). Then **commit and merge to `develop` at each milestone** via the merge protocol (rules 18–19, `merge-protocol`) — never hoard work in one long-lived branch dropped at the end. Integrating at every milestone keeps progress visible, conflicts small, and each milestone green and deployable.
21. **Role resolution — never guess your role.** If you **don't know your role**, or you **lack the skill/knowledge** for an assigned job → follow the **`role-resolution`** skill: **ask** the user (questioning-protocol) + **discover** the right capability (board `list_skills`/`get_skill`, Oriel `oriel_list_skills`/`oriel_get_skill` + its agents/expertise/prompts/mcp surfaces, plugins; honor `get_skill_sources` priority) + **load** it (`get_skill`/`adopt_oriel_skill`), then confirm scope **before** working. Never build without the needed skill.
22. **Two-tier spawn.** Spawn by tier: **user-facing roles** (TL, Designer, PM — anything needing face-to-face user discussion: interview/demo/decisions) are spawned as a **chip → NEW CHAT**; **doer roles** (Developer/Frontend/Backend/Data, Tester/QA, DevOps, Reviewer, Security, Writer) are spawned as **HEADLESS background sub-agents under a TL**, who speaks for them to the user. Doers never talk to the user directly.
23. **Self-driving loop.** Lead chats (PD/TL) **re-arm a ScheduleWakeup each tick** to self-drive the work to completion unattended (`check_in`-gated, so quiet ticks stay near-free). Escalate to the user **only** for genuine decisions / approvals.
24. **Continuous QA.** Before closing any epic/project, run the delivery gates in order: **contract → smoke → user-walk (`demo-e2e-walk`) → E2E-if-needed**. Don't skip the cheaper gates or jump straight to E2E.
25. **Prefer the best available skill — board + superpowers + plugins.** For your role/task, don't rely **only** on the board's skill: if a **superpowers** plugin skill (or another installed Claude plugin skill) covers the same job, **READ BOTH** the board skill and the plugin one and use the best — or combine them. Concrete examples to look for: **`superpowers:systematic-debugging`** (debugging), **`superpowers:test-driven-development`** (contract/TDD), **`superpowers:brainstorming`** + **`superpowers:writing-plans`** (design/planning), **`superpowers:verification-before-completion`** (before claiming done). This complements the **`role-resolution`** skill's discovery step (rule 21).
26. **Repo-agnostic board (via MCP).** The Agents Board is your **coordination** layer, reached **only** through the **Agents Board MCP** — it is **repo-agnostic**. Run the project in **whatever repository / working directory** the user needs (the current cwd, another repo, a brand-new one, or even a non-code initiative). You do **not** need to be in, clone, or modify the board's own codebase to use it — every board action is an **MCP tool call** (`create_project`, `create_entity`, `check_in`, `set_status`, …), **never** an edit to the board repo. For the actual **build**, adopt the **TARGET** repo's own conventions: read its `CLAUDE.md` / `AGENTS.md` / `README`, ask the user, or use the **`role-resolution`** skill to find the right skills/stack. **The build/test/deploy specifics in these conventions** (the safe-merge protocol, `get_migration_number`, pgTAP via the IPv4 pooler, worktree-from-`origin/develop`, `check:no-color`, the pre-push green-gate hook, the dev/UAT/Koyeb tiers — rules 1, 5, 10, 18, 19) are the **agent-board-mcp repo's OWN conventions**, shown as an **EXAMPLE**; for any other repo, **discover and follow THAT repo's conventions instead**.
